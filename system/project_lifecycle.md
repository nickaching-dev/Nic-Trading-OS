# Project Lifecycle

This document defines how projects move through NicOS.

Projects are temporary workspaces. They should have a beginning, active execution, and a clear ending state.

## Idea

The project exists as a possible initiative, but it is not yet defined clearly enough to start.

A project should stay in `Idea` when:

- the goal is still vague
- the scope is unclear
- the timing is not decided
- the owner has not committed to working on it

Move a project out of `Idea` when there is enough clarity to define purpose, goal, and success criteria.

## Planning

The project is being shaped into something actionable.

A project should be in `Planning` when:

- the purpose is clear
- the vision is becoming concrete
- milestones are being outlined
- risks and open questions are being identified

Move a project from `Planning` to `Active` when real execution begins.

## Active

The project is currently being worked on.

A project should be in `Active` when:

- next actions are clear
- work is ongoing
- progress is being updated
- decisions are actively being made

Move a project from `Active` to `Waiting` when progress is blocked or intentionally paused.

Move it to `Completed` when the goal and success criteria are met.

## Waiting

The project is still alive, but progress is paused.

A project should be in `Waiting` when:

- the next step depends on someone or something else
- the project is paused intentionally
- timing is not right for active work

Move a project back to `Active` when the blocker clears and meaningful execution resumes.

Move it to `Archived` if the project is no longer relevant and will not resume.

## Completed

The project has reached its intended end state.

A project should move to `Completed` when:

- the goal was achieved
- the main deliverable exists
- the core work is done
- the remaining value is mostly in lessons or reference material

Before moving a project to `Completed`, capture:

- important decisions
- lessons learned
- knowledge generated
- anything that should be distilled into the right module

## Archived

The project is inactive and no longer part of current execution.

A project should move to `Archived` when:

- it is no longer relevant
- it was replaced by another project
- it will not resume
- it is completed and no longer needs active visibility

Archiving does not mean forgetting. It means the project is no longer active, but its record is preserved.

## Final Rule

Projects should not become permanent knowledge dumps.

When a project finishes, the lasting knowledge should flow back into the right module.
